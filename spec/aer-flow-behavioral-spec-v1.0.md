# AER Flow Behavioral Specification — v1.0

This document is the authoritative definition of what AER Flow guarantees. Code is derived from this; this is not derived from code. It assumes AER Core v1.1 (see `aer-core-behavioral-spec-v1.1.md`) as a dependency and does not redefine anything Core already guarantees.

---

## 1. System Identity

AER Flow is a deterministic, event-sourced workflow controller built on top of AER Core.

It coordinates untrusted local processes (LLMs, CLIs, compilers, scripts) — represented as **workers** — by emitting `ExecutionRequest`s to Core and reacting to their outcomes.

Flow is **not** a workflow programming language. It has no loops, no conditional syntax, no runtime logic of its own. Workflow shape is declared statically; Flow only decides, at any given moment, what is ready to run.

> If a feature would not make sense for `git`, `cargo test`, or `ffmpeg`, it does not belong in Core.
> If a feature requires knowing what a workflow is, it does not belong in Core — it belongs here.

---

## 2. System Layers

```
WorkflowDefinition (Static DAG Config)
        ↓
Flow (Reactive Controller)        ← this document
        ↓
Event Store (Durable History)
        ↓
Core (Process Executor)            ← aer-core-behavioral-spec-v1.1.md
        ↓
Operating System
```

Flow never executes a process directly. It only ever does two things: read the Event Store, and emit `ExecutionRequest`s to Core.

---

## 3. ExecutionRequest (Shared Primitive)

The immutable unit of execution, shared by identity across Flow's and Core's halves of the Event Store.

```
ExecutionRequest
 ├── ExecutionId        (globally unique; generated once, by Flow)
 ├── WorkflowId
 ├── StepId
 ├── Worker              (logical execution target, e.g. "claude", "agy", "git", or a non-process party such as "human" — see §17)
 ├── Inputs               (file paths)
 ├── Outputs              (file paths)
 ├── Timeout
 ├── Environment
 └── UpstreamExecutionIds (map<StepId, ExecutionId> — see below)
```

**Properties:**
- Immutable once emitted. Never mutated, never reused.
- `ExecutionId` is generated by Flow and is the sole join key between Flow-owned and Core-owned events for a given execution. No other cross-referencing mechanism (timestamps, wall-clock proximity, file order) may be used to associate a Flow event with a Core event.
- Represents both intent (what should run) and enforcement parameters (timeout, environment) in one frozen record. Core neither knows nor needs to know the workflow this request belongs to — `WorkflowId`/`StepId` are opaque metadata to Core, carried only for Flow's own later replay.
- Not every `Worker` corresponds to a Core-managed process — see §17 for the case where the "execution" is satisfied by an external party rather than a spawned process.
- `UpstreamExecutionIds` records, for each `StepId` this step `DependsOn` (§11.1), exactly which of that dependency's `ExecutionId`s this request's `Inputs` were derived from. This is what makes staleness (§11.3, §17.5) a fact Flow can derive purely by reading the log, rather than something requiring its own tracked state: an execution is current with respect to a given dependency exactly when the dependency's latest successful `ExecutionId` still matches what's recorded here.

## 4. Worker Contracts

A **Worker** is a logical execution target (e.g. `claude`, `agy`, `git`) bound to a typed contract, not a vendor name.

```
WorkerContract
 ├── WorkerName
 ├── RequiredInputs[]    (named file roles, e.g. "goal", "plan-to-review")
 ├── ProducedOutputs[]   (named file roles, e.g. "plan", "review")
 └── OptionalMetadata[]  (e.g. a structured .json summary alongside the primary artifact)
```

**Rules:**
- A `WorkflowDefinition` step declares which `WorkerContract` it requires, not which vendor fulfills it. `Worker: "architect"` resolves to a concrete binary (`claude`, `agy`, future tools) via configuration external to the workflow — swapping the binding does not require editing the workflow.
- Flow validates that an execution's declared `Outputs` satisfy its `WorkerContract`'s `ProducedOutputs` before classifying the execution as successful (see §8). A worker that exits 0 but fails to produce a contractually required output is a failure, not a success.
- Contracts are declarative metadata only. They impose no runtime logic on Core — Core still only sees `Inputs`/`Outputs` as file paths from §3.
- `OptionalMetadata` is where a worker may report `FailureClassification` (§8.1) on failure. Flow understands exactly that one field, with exactly two values, and nothing else a worker writes there — see §8.1 for why this is deliberately narrow.

## 5. Event Store (Durable History)

Append-only, source of truth for system state reconstruction. Storage backend (JSONL, SQLite, etc.) is implementation-defined and is **not** part of this contract — only the guarantees below are.

### 5.1 Dual-log ownership

| Log                            | Owner             | Contains                                                                                                                                                                                                                              |
| ------------------------------ | ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `events.jsonl` (or equivalent) | Core, exclusively | `ExecutionStarted`, `ExecutionExited`, `StdoutChunk`/`StderrChunk` (if `CaptureOutput` is enabled)                                                                                                                                    |
| `flow.jsonl` (or equivalent)   | Flow, exclusively | `ExecutionRequestAccepted`, `ExecutionRequestRejected`, `ExecutionSucceeded`, `ExecutionFailed`, `ExecutionCancelled`, `CancellationRequested`, `WorkflowPaused`, `ExternalDecisionRecorded`, `WorkflowResumed`, `WorkflowTransition` |

**Rule:** each log has exactly one writer role. Core never writes to the Flow log; Flow never writes to the Core log. This holds regardless of file format or count — if the two are later merged into one store, the ownership rule still applies per event type.

### 5.2 Event naming and lifecycle semantics

| Event                                                             | Meaning                                                                                                 | Timing                                                                            |
| ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `ExecutionRequestAccepted`                                        | Flow has admitted this request for execution                                                            | Pre-execution (admission control)                                                 |
| `ExecutionRequestRejected`                                        | Flow declined to submit this request (e.g. concurrency cap)                                             | Pre-execution (admission control)                                                 |
| `ExecutionSucceeded`                                              | Flow has classified a completed execution as successful                                                 | Post-execution (outcome classification)                                           |
| `ExecutionFailed`                                                 | Flow has classified a completed execution as failed                                                     | Post-execution (outcome classification)                                           |
| `ExecutionCancelled`                                              | Flow has classified a completed execution as cancelled (§9)                                             | Post-execution (outcome classification)                                           |
| `CancellationRequested`                                           | Flow has forwarded an on-demand cancellation request toward Core for a still-running `ExecutionId` (§9) | Mid-execution (neither admission nor outcome — the execution was already running) |
| `WorkflowPaused` / `ExternalDecisionRecorded` / `WorkflowResumed` | A workflow has paused, received an external decision, and resumed (§17)                                 | Between steps, at arbitrary real-world duration                                   |

These names are deliberately distinct from each other precisely because "admitted to run," "succeeded once run," "cancelled mid-run," and "paused awaiting a decision" are different moments. No event name may be reused across multiple meanings.

### 5.3 Properties

- Append-only. Lines are never edited or deleted, including on retry — see §10.
- Fully replayable: given the full event history, the current workflow state is always reconstructable.
- The Event Store — not `manifest.json`, not any in-memory cache — is the absolute source of truth.
- **Appends are atomic at the line boundary.** A reader observing either log concurrently with a writer must never be able to read a partially-written event line — only complete lines, or nothing yet. This is a producer-side obligation (e.g. write-then-rename, or buffer-and-flush-on-newline) and applies regardless of who or what is reading: a replay on restart, a live log-tail, or a future watcher. It exists so that any reader can safely tail the log without coordinating with the writer.

## 6. Causal Linking

Because `events.jsonl` and `flow.jsonl` are independently written and have no shared monotonic counter or synchronized clock, they are interleaved during replay by structural reference, not wall-clock proximity.

**Rule:** every event in either log carries the `ExecutionId` (§3) it pertains to. Replay reconstructs the relationship between a Flow intent and its Core execution by matching on that shared ID — never by file order, line number, or timestamp comparison across the two logs.

If a Core execution exists in `events.jsonl` with no corresponding `ExecutionSucceeded`/`ExecutionFailed`/`ExecutionCancelled` in `flow.jsonl`, the projection concludes: the process ran, but Flow has not yet finalized its classification (either still in progress, or Flow crashed before recording the outcome).

---

## 7. Crash Durability & Write Sequence Rule

To guarantee idempotent recovery after a crash or power loss, writes obey a strict, non-negotiable order:

1. **Intent first.** Flow appends `ExecutionRequestAccepted` (with the new `ExecutionId`) to `flow.jsonl`.
2. **Hard flush.** That write is synchronously fsync'd before proceeding. Lifecycle-tier events (admission, outcome, cancellation request) are always sync-written; high-volume Tier 2 observation data (`StdoutChunk`/`StderrChunk`, owned by Core) may be buffered/async — see AER Core spec §4.
3. **Then, and only then,** Flow submits the `ExecutionRequest` to Core, which appends `ExecutionStarted` to `events.jsonl`.

**Rule:** Core must never be asked to execute a request before its intent is durably recorded. If the system crashes between steps 1 and 3, the resulting state — intent recorded, no execution trace — is the *expected*, safe crash state. Recovery simply re-submits the recorded intent. The reverse ordering (spawn before intent is durable) is forbidden: it risks an executing or already-exited process with no record of why it ran, which the causal-linking model in §6 has no way to attribute.

This same intent-first discipline governs cancellation (§9): `CancellationRequested` is recorded and fsync'd before the cancellation request is forwarded to Core, for the identical reason — and is safe to re-issue on crash recovery, since Core treats cancelling an already-finished execution as a no-op (AER Core spec §7).

## 8. Failure Model

Core reports only factual, mechanical outcomes — exit code and `reason` (`NaturalExit` | `TimedOut` | `CancelRequested`; see AER Core spec §2.3) — nothing about why a worker behaved as it did beyond that. Flow alone interprets these into one of three terminal classifications:

| Classification       | Condition                                                                                                               |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `ExecutionSucceeded` | Core `reason == NaturalExit`, exit code `== 0`, **and** all outputs declared by the step's `WorkerContract` exist       |
| `ExecutionFailed`    | Core `reason == NaturalExit` and (non-zero exit code **or** missing required outputs), **or** Core `reason == TimedOut` |
| `ExecutionCancelled` | Core `reason == CancelRequested` — see §9                                                                               |

A clean exit with missing outputs is a failure, not a success — exit code alone is necessary but not sufficient. A cancellation is never classified as a failure: it reflects an explicit decision to stop, not the worker doing anything wrong, and — per §9 and §10 — is never subject to retry.

### 8.1 Worker Self-Reported Failure Classification (optional)

When an execution is classified `ExecutionFailed`, a worker may optionally report its own judgment of whether retrying would help, via `OptionalMetadata` (§4):

```
FailureClassification: Permanent | Retryable
```

This is the *only* vocabulary Flow itself understands for this purpose — two fixed values, regardless of how many different vendors or tools become workers. Flow does not parse, store, or act on vendor-specific failure detail (e.g. `"auth_expired"`, `"merge_conflict"`, `"syntax_error"`); a worker remains free to emit as much of that detail as it likes elsewhere in its metadata, for a human or a log reader to act on, but only the two-value classification above changes Flow's own scheduling behavior — exactly as `RetryPolicy` (§10) already deals only in attempt counts and never in vendor semantics. This is deliberately narrower than an open-ended tag set: a freeform string becomes a de facto API the moment Flow's behavior depends on its exact spelling, and a worker's later rename of `"auth_expired"` to `"authentication_expired"` would silently change retry behavior if Flow ever matched on the string itself.

**Effect:**
- `Retryable` (the default if the field is absent or unrecognized): standard `RetryPolicy` behavior is unaffected — see §10.
- `Permanent`: Flow transitions the step directly to terminal failure, regardless of remaining attempts under `RetryPolicy`. This exists so a worker that knows with certainty that retrying is pointless (e.g. an expired credential no amount of re-running will fix) doesn't have to exhaust a multi-attempt policy purely to rediscover, on every attempt, what it already knew on the first one.

## 9. Cancellation

A live execution may be stopped on demand — distinct from a timeout, which is Core acting on a deadline it was configured with in advance. Cancellation is always externally initiated, at a moment Core cannot predict.

1. **Intent recorded first**, per §7's write sequence rule: Flow appends `CancellationRequested` (referencing the target `ExecutionId`) to `flow.jsonl` and fsyncs, before forwarding the request to Core's cancellation mechanism (AER Core spec §7 — the specific API shape is unspecified at this behavioral level; only the capability is required).
2. Core performs its existing kill sequence and process tree cleanup (AER Core spec §5–§6) and emits `Exited { reason: CancelRequested }`.
3. Flow observes that event via the causal link (§6) and appends `ExecutionCancelled` to `flow.jsonl`.
4. A cancellation request for an execution that has already reached a terminal outcome (`ExecutionSucceeded`/`ExecutionFailed`/`ExecutionCancelled`) is a no-op: Flow records that the request arrived too late and does not retroactively alter the already-recorded outcome.

**Rule:** `ExecutionCancelled` is a third terminal outcome alongside `ExecutionSucceeded`/`ExecutionFailed` (§8) and is never subject to `RetryPolicy` (§10). Cancellation reflects a decision to stop, not a failure to recover from.

If the system crashes between steps 1 and 2, the same idempotent recovery logic as ordinary execution applies (§7): on restart, Flow sees a recorded intent to cancel with no corresponding terminal Core event yet, and simply re-issues the cancellation request — safe regardless of whether the original request actually reached Core, because Core's handling of cancelling an already-exited execution is itself defined as a no-op.

Cancellation is requested through Flow's mutation interface (§14), like every other external action — it has no separate channel and no special privilege.

## 10. Retry Semantics

- A retry is a brand-new `ExecutionRequest` with a brand-new `ExecutionId`. The prior `ExecutionId` and its events are never mutated, edited, or removed.
- History therefore shows every attempt:
  ```
  ExecutionRequestAccepted  (ExecutionId: A, attempt 1)
  ExecutionFailed           (ExecutionId: A)
  ExecutionRequestAccepted  (ExecutionId: B, attempt 2)
  ExecutionSucceeded        (ExecutionId: B)
  ```
- `RetryPolicy` (declared per-step in `WorkflowDefinition`, §11) governs whether a failure triggers a new `ExecutionRequest` or transitions the step to terminal failure — subject to the `Permanent` short-circuit in §8.1.
- `ExecutionCancelled` (§9) is never retried, regardless of `RetryPolicy`. A cancellation is an instruction to stop, not a failure Flow should attempt to route around.

### 10.1 Retry as Bounded Self-Iteration (Pattern)

Flow has no loop or branching construct (§20). This section names a pattern — not a new primitive — for the common case of "have a single step retry itself until its own output is good enough, capped at a fixed number of attempts," using only mechanisms already defined in this document.

**The pattern:**
- A `WorkerContract` (§4) defines what counts as a satisfying output not just by *existence* of a file, but by its *content meeting a declared condition* — e.g. requiring an output `verdict.json` whose `status` field equals `"approved"`, not merely requiring `verdict.json` to exist.
- A worker that judges its own output insufficient (whether by self-critique, a second model call, or any other means internal to the worker boundary — §18.2) writes `verdict.json: {"status": "needs_revision", ...}` and exits 0.
- Per §8's failure model, exit code `0` is necessary but not sufficient for `ExecutionSucceeded` — the declared output must also satisfy the contract. `status: "needs_revision"` fails the contract, so this execution is classified `ExecutionFailed` despite the clean exit.
- The step's existing `RetryPolicy` then governs exactly as it would for any other failure: a new `ExecutionRequest` with a new `ExecutionId` is generated, up to the policy's attempt limit, after which the step transitions to terminal failure per the normal retry rules above.
- A worker using this pattern may additionally report `FailureClassification: Permanent` (§8.1) alongside `status: "needs_revision"` if it concludes further iteration won't help — stopping the loop immediately rather than exhausting the full attempt cap to rediscover the same conclusion repeatedly.

**What this buys, and what it doesn't:**
- This gets "iterate on a single step until it's good enough, bounded by a maximum attempt count" entirely from existing mechanism — no DSL, no new event type, no change to §20's exclusions. Every attempt is a fully independent, fully replayable `ExecutionRequest`, exactly like any other retry.
- It does **not** provide a way to route back to an *earlier* step in the DAG on its own. That shape requires either Case 2 encapsulation (§18.2, the loop lives inside one worker), explicit unrolling in the template (separate, named steps for each anticipated round), or the bounded `Supersede` mechanism (§17.2, §17.5) — a step declaring `SupersedeTargets` may explicitly, and only within that declared list, trigger a new execution of an earlier step. Retry-as-self-iteration itself still only loops a step in place against itself; it cannot reach backward — that capability exists elsewhere, deliberately bounded, not here.
- The cap on attempts is a real constraint, not a formality: a worker that can never reach `"approved"` will exhaust `RetryPolicy` and fail the step terminally, exactly as designed. This is intentional — an unbounded self-improvement loop is exactly the kind of runtime logic §20 excludes Flow from providing, and bounding it is what keeps this a pattern rather than a reintroduced DSL.

## 11. WorkflowDefinition

### 11.1 Template (Static DAG)

Declarative structure only — no loops, no conditionals, no runtime logic.

```
WorkflowDefinition (Template)
 ├── WorkflowTemplateId
 ├── WorkflowTemplateVersion
 └── Steps[]
      ├── StepId
      ├── Worker           (a WorkerContract name, §4 — not a vendor)
      ├── Inputs
      ├── Outputs
      ├── DependsOn[]      (StepIds)
      ├── RetryPolicy
      └── PausePoint       (optional — see §17)
           └── SupersedeTargets[]   (StepIds; optional — see §17.1, §17.2)

```

A `WorkflowDefinition` in this form is a **template**: editable, versionable, and not itself bound to any running task. `WorkflowTemplateVersion` increments on every edit that is instantiated from; it is the only thing that distinguishes two otherwise-identically-named templates over time.

### 11.2 WorkflowDefinitionSnapshot (Immutable Binding)

When a task is created, Flow binds it to an immutable `WorkflowDefinitionSnapshot` — a frozen copy of the template as it existed at that moment.

```
WorkflowDefinitionSnapshot
 ├── WorkflowDefinitionSnapshotId   (globally unique; generated once, by Flow)
 ├── WorkflowTemplateId             (which template this was instantiated from)
 ├── WorkflowTemplateVersion        (which version of that template, at instantiation time)
 └── Steps[]                        (frozen copy of the template's Steps at that moment)
```

```
Template (id, version N)  ──edit──▶  Template (id, version N+1)

Template (id, version N, at instantiation time)
        │
        ▼ instantiate
WorkflowDefinitionSnapshot (new SnapshotId, recording TemplateId + version N)
        │
        ▼
       Task
```

**Rules:**
- A running or historical task is permanently bound to the `WorkflowDefinitionSnapshot` it was created from. Editing the template after task creation has no effect on that task, in progress or in replay.
- `WorkflowDefinitionSnapshotId`, `WorkflowTemplateId`, and `WorkflowTemplateVersion` are first-class, queryable metadata — not derived or reconstructed after the fact. This is what makes "which template produced this snapshot, and which version" an answerable question without needing to diff file contents.
- Every reference to "WorkflowDefinition" elsewhere in this document, including the Determinism Guarantee (§13) and Dependency Resolution Rule (§11.3), means the task's bound `WorkflowDefinitionSnapshot` — never "whatever the current template says."
- Flow never mutates or patches a `WorkflowDefinitionSnapshot` once a task is bound to it. A workflow change for an in-flight task is only possible by defining a new template version and creating a new task from it — never by altering history's binding.
- This is what makes §13's determinism guarantee well-defined in the presence of editable templates: "identical WorkflowDefinition" is only a coherent claim because the snapshot, not the live template, is what Flow actually reads.

### 11.3 Dependency Resolution Rule (critical)

`DependsOn` references `StepId`, never `ExecutionId`. A step is ready to run when, for **every** `StepId` it depends on:

1. that dependency's most recent `ExecutionRequest` has produced `ExecutionSucceeded`, **and**
2. this step does not already have a successful execution whose `UpstreamExecutionIds` entry for that dependency matches the dependency's current most recent successful `ExecutionId`.

Condition 1 is the original rule, unchanged. Condition 2 is what makes a step eligible to run *again* after it has already succeeded once: if a dependency has since produced a *newer* successful execution (via `Supersede`, §17.2) than the one this step actually used, this step's prior success no longer reflects current upstream state, and it becomes ready to run once more — see §17.5.

Earlier, failed or cancelled attempts for the same `StepId` are part of history (§10) but are never re-examined for scheduling purposes — only the latest attempt per step matters for readiness. If the most recent attempt is paused awaiting an external decision (§17) rather than terminal, the step is not yet ready and remains pending until that decision resolves.

A step whose most recent successful execution satisfies condition 1 but fails condition 2 is referred to in this document as having a **stale** execution with respect to that dependency — a derived fact, not a stored or event-bearing one (§17.5).

## 12. Flow as a Projection Function

```
FlowState         = Project(EventStore, WorkflowDefinitionSnapshot)
NextExecutionRequest = f(FlowState)
```

**Responsibilities:**
- Reconstruct workflow state from event history (never from live process state, wall-clock time, or anything not frozen inside an event — see Determinism Guarantee, §13).
- Determine which steps are ready per §11.3.
- Emit `ExecutionRequest`s for ready steps.
- Classify completed executions per §8.
- Advance `WorkflowTransition`, `WorkflowPaused`/`WorkflowResumed` state accordingly.

**Constraint:** Flow never executes a process. It only ever reads the Event Store and emits requests.

### 12.1 manifest.json is a derived cache, not a second source of truth

If an implementation maintains a `manifest.json` (or equivalent) as a fast-read snapshot of current state, it **must** be reconstructable in full by replaying the Event Store from the beginning (or from the manifest's own last-recorded checkpoint plus the remaining tail of the log). Any field that cannot be derived purely from replaying events does not belong in the manifest — it belongs in an event. The manifest must be safe to delete and rebuild at any time with no information loss.

## 13. Determinism Guarantee

Given identical:
- Event Store contents, and
- `WorkflowDefinitionSnapshot` (§11.2), and
- Flow logic version,

Flow **must** produce an identical sequence of `ExecutionRequest`s, every time, with no dependency on wall-clock time, environment variables, or live filesystem state outside of what is frozen inside the event data itself.

This guarantees crash-recovery correctness, reproducibility, and full auditability.

---

## 14. Mutation Interface

Flow's state is mutated only through a single, well-defined interface. This document does not assume that any particular client is privileged over another, and does not name, require, or constrain a specific implementation of that interface.

A reference implementation exists today and is described in a separate document (the AER CLI specification) — but the CLI is not part of this contract. If a second implementation of the mutation interface is ever built (for example, an embedded library consumed by a UI directly, or a network-facing service), the shared interface should be extracted from what the two concrete implementations actually have in common at that time, not designed speculatively now for hypothetical future consumers. Until then, this document deliberately says no more about the interface's shape than "exactly one well-defined mutation surface exists." Every mutation discussed elsewhere in this document — submitting work, cancelling it (§9), or recording an external decision (§17) — goes through this same interface; none of them has a separate channel.

---

## 15. Concurrency Model

**Guarantee (behavior, not mechanism):** at most one Flow instance may mutate a given task's workflow state at a time. At most one Core execution exists per `ExecutionRequest`.

The implementation today may use a kernel-held advisory file lock (e.g. `FileShare.None` on a `FileStream`) scoped to the task namespace. Such a lock releases automatically the instant its owning process exits — a crashed holder never leaves a deadlock behind. A sentinel file whose mere *existence* signals "locked" does **not** satisfy this guarantee, because it survives a crash and must be manually cleared. Whatever the mechanism, this is the property that must hold; the mechanism itself is not part of the contract and may change.

## 16. Artifact Model

Workers are blind to versioning, lineage, and path topology. They receive only an input set of paths and one output directory; they do not choose either.

```
artifacts/
 ├── execution_891/
 │    └── plan.md
 └── execution_892/
      └── review.md
```

**Rules:**
- Flow computes and assigns all paths before a worker is invoked, and passes them in (e.g. as environment variables: `AER_INPUT_<n>`, `AER_OUTPUT_DIR`).
- Artifacts are immutable once an execution completes. A revision is a new execution with a new `ExecutionId` and a new output directory — never an edit in place.
- This is what makes data lineage fully reconstructable from the Event Store alone: every artifact's provenance is exactly the `ExecutionId` of the directory it lives in.
- This rule applies identically whether the producing "worker" is a Core-managed process or a non-process party such as a human (§17) — there is exactly one artifact model in this system, not a special case for human-produced files.

## 17. Pause & External Decision

A workflow may pause between steps to wait for a decision from outside the system, then resume. This generalizes what earlier revisions of this document called "Human Approval": approval is one *kind* of external decision, not the only kind, and the human reviewing a plan before the next step runs is the same primitive as a future automated policy check, a cost-approval gate, or any other producer of an external decision. Flow does not need a different mechanism for each.

```
ExecutionSucceeded / ExecutionFailed / ExecutionCancelled
        │
        ▼   (only if this step declares a PausePoint, §11.1)
WorkflowPaused
        │
        ▼   (zero or more decisions, over arbitrary real-world time)
ExternalDecisionRecorded
        │
        ▼
WorkflowResumed
```

### 17.1 PausePoint

A step in a `WorkflowDefinition` template (§11.1) may optionally declare a `PausePoint`. When a step with `PausePoint` reaches a terminal outcome, Flow appends `WorkflowPaused` instead of immediately evaluating downstream steps' readiness (§11.3). The workflow remains paused — Flow is idle, no process exists, nothing is "waiting" in memory — until an `ExternalDecisionRecorded` event resumes it.

A `PausePoint` may optionally declare `SupersedeTargets[]` — the set of earlier `StepId`s a `Supersede` decision made at this pause point may target. A `Supersede` decision names exactly one target, chosen from this list (§17.2) — not multiple, and not anything outside it.

**Rules:**
- Every `StepId` in `SupersedeTargets` shall be a transitive ancestor of the step declaring the `PausePoint` (reachable by following `DependsOn` backward from this step). Any other value is invalid. A target with no dependency relationship to this pause point could never be connected back to this step's own cascade (§17.5), so allowing it would grant a capability the template couldn't actually deliver.
- Every entry in `SupersedeTargets` shall be unique.
- A step shall never list itself in its own `SupersedeTargets` (already implied by "transitive ancestor," but stated directly rather than left to be derived).
- A `Supersede` may target **any single member** of `SupersedeTargets`, with no further restriction — including a member that is itself an ancestor of another member on the same list. This needs no additional rule because §17.5's cascade already propagates through such relationships automatically: targeting the more distant ancestor will, once it succeeds, make the nearer one stale in turn through the ordinary readiness check, exactly as it would if they'd been targeted one at a time.

This list lives on the pause point, not on the upstream step, so that everything which can happen at a given pause is visible in one place in the template, and adding a new revision path is a local edit to the pause point rather than a coordinated edit across two distant parts of the template. If `SupersedeTargets` is absent or empty, this pause point supports `Resume`/`Reject`/`RetryWithRevision` only — no `Supersede` target is available, and the template is unambiguous about that simply by the field being empty.

### 17.2 ExternalDecisionRecorded

```
ExternalDecisionRecorded
 ├── DecisionId
 ├── ReferencedExecutionId       (which execution's outcome this decision responds to)
 ├── DecisionType                 (Resume | Reject | RetryWithRevision | Supersede)
 ├── TargetStepId                 (required only for Supersede — see below)
 └── SupplementaryExecutionId    (optional for RetryWithRevision; required for Supersede — see §17.3)
```

`DecisionType` is deliberately a small, closed set that Flow itself acts on differently — not an open string, for the same reason `FailureClassification` (§8.1) is closed rather than freeform:

- **`Resume`** — proceed as if the referenced execution's outcome stands; downstream steps become eligible per §11.3 exactly as they would without a pause.
- **`Reject`** — treat the referenced step as terminally failed; do not evaluate its dependents. Equivalent in effect to exhausting `RetryPolicy`, but externally triggered rather than automatic.
- **`RetryWithRevision`** — the referenced step, which has **not yet succeeded**, should attempt again, optionally incorporating a supplementary artifact. This is the externally-triggered counterpart to §10.1's bounded self-iteration: same step, new attempt, recoverable failure. `SupplementaryExecutionId` is optional here — sometimes a human just wants to retry as-is.
- **`Supersede`** — a step that has **already succeeded** should be re-executed because a downstream step has learned something that makes the prior result no longer current. `TargetStepId` must be one of the pause point's declared `SupersedeTargets` entries — a `Supersede` naming any other `StepId` is invalid and must be rejected by Flow, not silently widened. `SupplementaryExecutionId` is **mandatory** for `Supersede`: by definition, superseding a result that already succeeded only makes sense if there's new information driving the change; a `Supersede` with no supplementary artifact has no reason to exist and is itself an invalid decision.

`RetryWithRevision` and `Supersede` are kept as separate decision types, rather than one type covering both, specifically so the event log itself records which relationship occurred — "this step hadn't succeeded yet and tried again" reads as a different fact from "this step had already succeeded and was deliberately superseded," and a log reader (or a human auditing history later) shouldn't have to infer which one happened from context.

Neither type, nor any other part of `DecisionType`, includes anything resembling "jump to a different, non-dependent step" — that would reintroduce arbitrary runtime branching, which §20 excludes for the same reasons a workflow DSL is excluded. `Supersede` is not an exception to that exclusion: it can only ever target a `StepId` the *template* already declared reachable from this exact pause point, ahead of time. The complete space of what can happen remains fully readable from the template alone, the same property `DependsOn` already provides for the forward graph. Changing *which worker* handles an upcoming step, independent of any of this, is a template-editing action (§4, and AER Flow UI spec §9), not a decision type Flow interprets at runtime.

### 17.3 Non-process external parties (e.g. a human supplying a revision)

When `RetryWithRevision` or `Supersede` involves a human (or any other non-process party) producing a new artifact, that act is itself modeled as an ordinary `ExecutionRequest`, exactly as already established for human approval in earlier revisions of this spec: *a human is a worker tier whose "execution" is an external event rather than a Core-managed process.*

Concretely:
- Flow mints a new `ExecutionId` and pre-allocates an output directory for it (§16), exactly as it would for any worker, with `Worker` set to a non-process value such as `"human"`.
- Flow appends `ExecutionRequestAccepted` for it, as for any other request.
- Because no Core process is spawned, there is no `Started`/`Exited` pair in `events.jsonl` for this `ExecutionId` — completion is instead detected by the contractually required output appearing at the assigned path, at which point Flow appends `ExecutionSucceeded` for it, exactly as §8 already defines for any worker satisfying its contract.
- `ExternalDecisionRecorded`'s `SupplementaryExecutionId` (§17.2) then references this `ExecutionId`.

**This resolves, by construction, the question of which exact version of an artifact a given decision applies to.** Because every artifact is immutable and addressed by the `ExecutionId` of the execution that produced it (§16), and a human-produced revision is itself such an execution, `ExternalDecisionRecorded` never refers to "the current state of a file" — only to a specific, frozen, already-immutable artifact. If a human edits a plan twice during a three-hour pause, there are simply two immutable artifacts in history, each under its own `ExecutionId`, and the recorded decision names exactly one of them. There is no live, mutable "current" version to be ambiguous about.

### 17.4 Scope boundary

This section specifies pausing *between* `ExecutionRequest`s only. Interrupting, redirecting, or supplying input to a worker *while it is actively running* — without stopping it first — is a different and substantially more expensive problem (Core Tier 3: stdin, PTY, interactive state) and remains explicitly out of scope; see §20 and AER project overview, §6. A worker may still expose its own internal progress for observation while running (via the Observation Tier — AER Core spec §2.2/§4, or via numbered artifacts written as it goes), and a running worker can always be stopped via Cancellation (§9) — but stopping is not the same as interactively steering a worker mid-execution, and this document does not provide the latter.

### 17.5 Supersession and the Automatic Invalidation Cascade

`Supersede` produces a new, successful execution for an earlier `StepId`. Everything that happens next is a direct consequence of §11.3's amended Dependency Resolution Rule — no separate cascade mechanism, event type, or scheduling rule is introduced here beyond that one amendment.

Concretely:

```
Architect succeeds (ExecutionId: A1)
        ↓
Critic runs, UpstreamExecutionIds: {Architect: A1}
        ↓
Critic succeeds, reaches its PausePoint with SupersedeTargets: [Architect]
        ↓
WorkflowPaused
        ↓
ExternalDecisionRecorded {
  DecisionType: Supersede,
  TargetStepId: Architect,
  SupplementaryExecutionId: <Critic's feedback artifact's ExecutionId>
}
        ↓
New ExecutionRequest for Architect (ExecutionId: A2),
including the supplementary artifact as an input
        ↓
Architect succeeds again (ExecutionId: A2)
```

At this point, nothing has explicitly told Flow to rerun Critic. But per §11.3, Critic's existing successful execution recorded `UpstreamExecutionIds: {Architect: A1}` — and Architect's *current* most recent success is now `A2`, not `A1`. Condition 2 of §11.3 is no longer satisfied for Critic with respect to Architect. Critic is therefore, again, a ready step: its own dependency (Architect) just succeeded, and Critic's last execution doesn't reflect that success. A new `ExecutionRequest` for Critic is generated through entirely ordinary scheduling — the same code path that would fire if Critic had never run at all.

If Critic also has a `PausePoint`, the cascade stops there exactly as it would on a first pass — a human reviews the new critique before anything downstream of Critic is touched. If Critic has no `PausePoint`, its successful rerun makes its own dependents stale in turn, and the cascade continues forward automatically, one `StepId` at a time, exactly following the graph that was already declared. **The cascade is automatically bounded by whatever `PausePoint`s already exist downstream** — nothing about `Supersede` changes how far an automatic rerun is allowed to travel unattended; that was already a property of the template, declared before any of this happened.

This is also why no new event type was needed: "is this stale" is fully derivable, on every read, from `ExecutionSucceeded` events and the `UpstreamExecutionIds` already recorded on each `ExecutionRequest` — exactly the same "derive, don't track" discipline already applied to `manifest.json` (§12.1). Staleness is a projection, not a fact Flow has to remember to write down.

## 18. Multi-Model Execution (Composition-Only Model)

Flow defines **no** multi-model primitive. Multi-model behavior is entirely emergent from composing standard `ExecutionRequest`s and workers, in exactly one of two shapes:

### 18.1 Case 1 — Sequential / Parallel DAG Composition (preferred)

Each model is its own `WorkflowDefinition` step:

```
Claude (architect)  →  Gemini (critic)  →  GPT (synthesizer)
```

Each is an independent `ExecutionRequest` with its own `DependsOn`. No shared runtime state exists between them beyond artifacts on disk. Any step in this chain may declare `PausePoint` (§17.1) to allow a human or other external party to review and decide before the next step runs — including, via `SupersedeTargets` and `Supersede` (§17.2, §17.5), sending feedback back to an earlier step and letting the rerun cascade forward automatically: `Architect → Critic`, Critic pauses with `SupersedeTargets: [Architect]`, a human supplies Critic's feedback via `Supersede`, Architect reruns, and Critic automatically reruns against the new plan once Architect succeeds again — without anything explicitly telling Flow to "go back." This is the direct realization of the original goal this project was built for: letting one model's critique drive another model's revision, with a human able to watch and steer the exchange at each step.

### 18.2 Case 2 — Encapsulated Multi-Model Worker

A single `ExecutionRequest` may internally coordinate multiple models. To Core and Flow, this is indistinguishable from running `cargo test` — one process, one lifecycle, one output directory. Internally, the worker may call multiple model CLIs, run multi-turn conversations, or perform iterative refinement; none of that is visible above the worker boundary.

**Rule:** if coordination between models requires shared state, loops, or interaction, that logic belongs *inside* the worker boundary — never inside Flow or Core.

**Explicit tradeoff (must be acknowledged, not discovered later):** encapsulated workers trade away fine-grained crash resumability. If the process crashes mid-worker — after invoking Claude, before invoking Gemini, say — Core only knows the process started and did not cleanly exit. There is no partial-progress recovery inside an encapsulated execution. A crash restarts the entire encapsulated execution from the beginning, not from its last internal step. This is an accepted cost of the simplicity Case 2 buys; it is not a defect to be fixed later. Pausing for a human decision *between* a Case 2 worker's internal turns (rather than only before or after the whole execution) falls under §17.4's scope boundary and is not provided.

## 19. System Principle

> If it would not make sense for `git`, `cargo test`, or `ffmpeg`, it does not belong in Core.
> If it requires knowing what a *workflow* is — steps, dependencies, retries, pauses — it belongs here, in Flow.

## 20. Explicitly Out of Scope

These are not gaps — they are deliberate exclusions, listed so they are not silently reintroduced later:

- **No workflow DSL.** No loops, no conditional syntax, no runtime logic in `WorkflowDefinition`. Control flow exists only as the consequence of observed execution outcomes feeding back into §11.3's readiness check, plus the bounded patterns in §10.1 and §17.5. `Supersede`'s bounded backward routing (§17.2) is not an exception to this: every step it can ever target is declared statically in the template's `SupersedeTargets` list, never chosen freely at runtime.
- **No daemon.** Flow is invoked as a process per mutation-interface call, not a persistent background service. Any future "watch and react automatically" capability is a different system built on top of this one, not a revision to this contract.
- **No cross-task event ordering.** Each task namespace's Event Store is independent. Flow makes no claim about relative ordering of events between two different tasks.
- **No named client architecture.** This document does not enumerate, privilege, or design around any specific client. See §14.
- **No interactive mid-execution control.** Pausing, deciding, and resuming (§17) happens only *between* executions. Interrupting or steering a worker while it is actively running is Core Tier 3 territory and is not specified here. See §17.4.
- **No open-ended `DecisionType` or `FailureClassification` vocabulary.** Both are deliberately small, closed sets (`Resume | Reject | RetryWithRevision | Supersede`, §17.2; `Permanent | Retryable`, §8.1). Expanding either is a real design decision to revisit only when a concrete case doesn't fit — not something to grow speculatively in advance.

## 21. Open Questions

- **Event Store performance at scale.** The per-task-namespace scoping keeps any single task's log small by construction, but no explicit strategy is committed yet for how Flow reads the log efficiently across many invocations (full reread each time vs. manifest-checkpoint-plus-tail). Deferred until the no-daemon question in §20 is revisited, since the efficient strategies differ depending on that answer.
- **Storage backend choice (JSONL vs. SQLite vs. other).** Explicitly not part of this contract per §5. Recommendation, not requirement: start with JSONL for human-readability and tooling-independence; revisit only if scale demands it.
- **Shape of the mutation interface itself (§14).** Deliberately unspecified beyond "exactly one." The CLI is today's reference implementation; nothing here commits to its API surface as the long-term contract.
- **Interactive mid-execution control (§17.4, §20).** Genuinely out of scope for now, not merely deferred wording — would require Core Tier 3 (stdin/PTY) and a substantially different worker-resumability model. Revisit only if a concrete use case needs steering a worker mid-run rather than only pausing between runs.
